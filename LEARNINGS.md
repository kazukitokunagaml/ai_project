# Continuous Learning Log (継続的学習ログ)

本ドキュメントはAIエージェントの「永続的メモリ」として機能する。各タスク完了時に、得られた教訓・エラーと解決策・設計判断の理由を記録し、同じ過ちの繰り返しを防ぐ。

---

## 記録フォーマット

新しいエントリは以下の形式で**ファイル末尾**に追記すること。

```markdown
## [YYYY-MM-DD] タスク: <タスクの概要>

### PRAR サイクルの要約
- **Perceive**: 何を認識・調査したか
- **Reason**: どのような推論・判断を行ったか
- **Act**: 何を実装・変更したか
- **Refine**: どのように検証・洗練したか

### 問題と解決策
- **問題**: 遭遇した具体的な問題
- **原因**: 根本原因の分析
- **解決策**: 採用した解決方法

### 設計判断
- なぜこの技術/パターンを選択したか
- 検討したが却下した代替案とその理由

### 教訓
- 今後同様のタスクで注意すべき点
- プロジェクト固有の知見
```

## [2026-02-17] タスク: AI向けコンテキスト・ドキュメントの初期構築

### PRAR サイクルの要約
- **Perceive**: プロジェクトは初期段階（`.gitignore` と `README.md` のみ）であることを確認。AI向けコンテキスト・エンジニアリングの調査結果を元に、3層構造のドキュメント体系を設計する必要があることを認識した。
- **Reason**: CLAUDE.md, GEMINI.md, AGENTS.md 等のベストプラクティスを統合し、エージェント非依存な汎用的ドキュメントとして `master.md`（憲法）、`rules.md`（詳細ルール）、`LEARNINGS.md`（学習ログ）の3層構造が最適と判断した。
- **Act**: 調査結果から外科的変更の原則、PRARワークフロー、境界条件(Always/Ask First/Never)、コーディング規約、テスト規約、セキュリティ規約を抽出し、3つのドキュメントを作成した。
- **Refine**: 各ドキュメント間の役割分担を明確にし、相互参照が機能するように構成した。

### 設計判断
- **3層構造を採用した理由**: 1つの巨大なファイルではトークン消費が大きくなり、また更新頻度の異なる情報を分離することで保守性を向上させるため
- **エージェント非依存にした理由**: Claude, Gemini, Copilot 等の特定ツールに依存せず、どのAIでも利用可能にするため
- **技術スタックを Python に設定した理由**: `.gitignore` の内容がPython向けに構成されており、プロジェクトの意図と一致するため

### 教訓
- AI向けドキュメントはプロジェクトの成長に合わせて継続的に更新する必要がある。技術スタックが確定した段階で `master.md` の技術スタックセクションと `rules.md` の言語固有ルールを更新すること
- `LEARNINGS.md` はAIだけでなく、人間の開発者にとっても有益なプロジェクト知識ベースとして機能する
